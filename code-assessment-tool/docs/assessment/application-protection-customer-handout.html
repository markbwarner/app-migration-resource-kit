Customer Guidance

Where Should Application Protection Be Implemented?

In most enterprise multi-tier applications, the best default place to implement protect and reveal APIs is usually the shared back-end service layer, not the UI layer.

Why Back End Usually Wins

Back-end services typically own validation, business rules, transformation logic, service endpoints, and access to corporate assets such as databases, Kafka, files, and downstream systems.

Why UI Still Matters

UI and client applications still matter as context because they collect, display, and sometimes directly submit sensitive fields, but they are not always the best long-term control point.

Why The Scanner Helps

The scanner helps separate front-end references from likely back-end or JDBC change owners so scope discussions are based on code evidence instead of assumptions.

Default Recommendation

For customer-facing multi-tier applications, protect and reveal APIs are usually best implemented in shared back-end service endpoints because that layer can enforce controls once and serve multiple channels consistently.

  • Less duplicated logic across multiple UIs
  • Better alignment with firewall and network boundaries
  • Easier governance, auditing, and policy consistency
  • Cleaner maintenance when security behavior changes

Important Exception

Some internal applications or thick-client style apps may already act as the real integration layer. In those cases, the client or UI-tier application may still be a valid place to integrate protect and reveal APIs.

  • No meaningful shared middle tier exists
  • The client directly invokes enterprise systems
  • The client owns workflow or data handling decisions
  • Protection must happen before data leaves the client boundary
The key question is not just where the field appears. The better question is which application layer truly owns the data-handling decision.

Scanner Output

What `frontend_to_backend_correlations` Adds

This section helps customers understand that front-end references do not automatically imply front-end implementation scope. It highlights likely back-end files that appear related through routes, payload fields, and naming overlap.

  • Reduces false scope
  • Shows likely ownership chains from UI to service layer
  • Supports conversations about where CRDP work is most likely to belong

It is heuristic guidance, not a full runtime dependency graph.

Most Useful Report Fields

  • likely_change_target
  • recommended_change_action
  • likely_change_owner
  • ownership_confidence
  • backend_owner_confidence
  • endpoint_correlation_score
  • code_change_candidate_count
  • jdbc_candidate_count
  • complexity_rating and complexity_score

Best First Filter

The easiest review set is usually the files where:

  • likely_change_target = true
  • recommended_change_action = review_crdp_rest_change
  • recommended_change_action = review_jdbc_substitution
  • recommended_change_action = review_data_access_change

Best DBA Handoff Fields

  • jdbc_tables
  • sensitive_columns
  • sensitive_tables
  • generated DBA planning SQL file

These outputs help DBAs review table structure, current max stored lengths, and possible width impacts before rollout.

Summary

A front end may show five sensitive fields, but that does not automatically mean five front-end code changes are required. In many cases, those fields flow through shared back-end web services that already own validation, orchestration, and persistence. That shared back-end layer is often the best place to implement protection and reveal APIs because it reduces duplication, centralizes control, and protects multiple channels at once.

Good Fit For Back-End Integration

  • Customer-facing multi-tier applications
  • Shared service endpoints already exist
  • Business rules live in the service layer
  • Corporate assets are accessed from the back end
  • One implementation should serve multiple channels

Good Fit For Client or UI Integration

  • No meaningful shared middle tier exists
  • Internal app already acts as the integration client
  • Local workflow owns the data-handling decision
  • Protection must occur before data leaves the client boundary
The scanner helps customers move from “where is sensitive data visible?” to “which layer most likely needs to change?”
