Application Protection Decision Matrix

Use this matrix to explain where protect and reveal APIs are usually best implemented, and how to interpret scanner output when deciding whether the UI or the back end is the better control point.

Architecture Decision Matrix

Scenario Typical Architecture Best Default Integration Point Why
Customer-facing multi-tier web app React, Angular, or Node.js UI calling shared APIs Shared back-end service endpoint Centralized enforcement, less duplication, better security boundary
Internal multi-tier app UI plus reusable service layer Shared back-end service endpoint Same control and reuse advantages still apply
Internal app acting as real integration client UI directly invokes enterprise services and owns workflow Client or UI may be valid The client may already be the effective integration layer
Back-end batch or service app Java, .NET, Python, Go, or Node.js service with no UI Back-end app itself It already performs the real data movement and transformation
Kafka producer or consumer flow Service publishes or consumes sensitive events Producer or consumer service Protection should happen where the event is created, transformed, or revealed
Database-heavy service Service uses JDBC or SQL to persist sensitive data Data-access layer or JDBC approach May be a strong JDBC-driver candidate with lower code impact
Shared enterprise API used by many channels Web, mobile, internal tools call the same API Shared API or service layer Highest reuse and best governance point

Criteria Matrix

Criteria Favors Back-End Implementation Favors UI or Client Implementation
Multiple UIs use the same service Yes No
Shared service endpoints already exist Yes No
Business logic lives in the service layer Yes No
Database, Kafka, file, or MQ access happens in the back end Yes No
Need centralized audit and policy enforcement Yes Sometimes
UI mostly collects and displays data Yes No
Client already acts as the integration layer Sometimes Yes
No reusable middle tier exists No Yes
Need one implementation reused across channels Yes No
In most enterprise multi-tier applications, the shared back-end service or integration layer is usually the best default implementation point. The scanner’s `frontend_to_backend_correlations`, `likely_change_target`, and `recommended_change_action` fields help show when front-end references point to a smaller and more realistic set of back-end owners.
