Application Code Sensitive Data Scanner Assessment Tool

A source-code assessment accelerator that finds sensitive field references, identifies likely change owners, and highlights where a Thales JDBC driver approach may reduce or eliminate application code changes.

Scope Clarity Separates broad sensitive-field awareness from the files that are more likely to require implementation change.
Faster Planning Helps architects and project teams reduce guesswork in level-of-effort, workshop prep, and migration scoping.
Actionable Output Provides repeatable console and JSON reporting for developers, delivery teams, and DevOps teams.

Why It Matters

Customers generally know they need to protect sensitive data. The harder question is where the real migration effort lives. Multi-tier applications often reference sensitive fields in front-end code, but the real implementation work usually sits in back-end services, API layers, JDBC paths, SQL statements, Kafka flows, and other integration logic.

This scanner gives customers a fast, explainable way to move from architecture assumptions to code-based evidence.

The scanner is built for source code, not production data values. It is designed to find field names like `ssn`, `dateOfBirth`, `acctNbr`, `householdId`, and `salaryAmount` across real codebases.
Questions It Answers

Where do sensitive fields appear, which files are likely front-end only, which files look like true back-end owners, and where JDBC may lower impact.

Typical Application Tiers

React, Angular, or Node.js clients; Spring Boot, .NET, Python, Go, or Node.js services; and JDBC, SQL, Kafka, or vector-search integrations.

Planning Value

Takes guesswork out of migration discussions by showing where testing, delivery effort, and implementation ownership are likely to concentrate.

Executive Summary And How The Scanner Works

Code-Aware Identifier Discovery

The scanner extracts identifier-like tokens, normalizes naming styles, and compares them to built-in and customer-defined aliases.

  • Raw token: accntNbr
  • Normalized token: accnt_nbr
  • Configured alias: accnt_nbr
  • Result: CUSTOM_ACCOUNT_NUMBER

This works well across camelCase, PascalCase, snake_case, and customer-specific abbreviations such as acctNbr, accntNbr, householdNbr, and hhId.

Ownership And Route Correlation

The scanner now goes beyond field counting. It looks for route ownership, API layers, SQL and JDBC access, and supporting DTO/model files.

  • frontend_reference_only
  • backend_logic_owner
  • data_access_owner
  • jdbc_candidate
  • supporting_model

In this context, a DTO is a Data Transfer Object: a simple class or record used to carry data between layers, such as a request body, response payload, or model shape passed between controller, service, and repository code. It usually describes structure, not business logic.

What It Produces

Line-level findings, matched attributes, category summaries, progress indicators, and optional JSON output.

What It Classifies

Front end, back end, data access, likely change owner, role in flow, and supporting model patterns.

Why It Helps

Shows the difference between files that mention sensitive data and files that are more likely to require CRDP or JDBC migration work.

Metrics, Complexity, And Customer Outcome

Files Scanned

Total files analyzed in scope, including source and related code artifacts.

Files With Indicators

Files containing at least one built-in or customer-defined sensitive field reference.

Total Matches

Line-level findings across the codebase. Useful for sizing review effort, not just implementation effort.

JDBC Candidates

Estimated areas where SQL and JDBC evidence suggest a lower-impact migration path.

Code-Change Candidates

Estimated service and orchestration areas where CRDP REST changes may require implementation work.

Ownership Fields

Likely change owner, role in flow, related files, endpoint correlation, and system-of-record indicators.

Complexity Index Inputs

  • Lines of code
  • Number of sensitive-field matches
  • REST and orchestration markers
  • SQL and JDBC markers
  • Likely JDBC substitution opportunity
  • Layer and ownership classification

Complexity Ratings

  • Low: limited touchpoints or mostly front-end-only awareness
  • Medium: bounded implementation effort with manageable testing scope
  • High: multiple sensitive fields, integrations, and larger regression scope

The complexity index is a planning aid. It is not a replacement for architecture review, but it helps teams start with evidence instead of guesswork.

What The Executive Summary Adds

  • Likely change-owner counts across the scanned portfolio
  • Role-in-flow counts to separate collection, transformation, and persistence
  • Top front-end to back-end and back-end to data-access correlations
  • JDBC table and sensitive-column summaries for DBA handoff

Who Benefits

  • Architects and project planners sizing migration effort
  • Developers focusing on likely implementation owners
  • DBAs validating table and column scope earlier
  • DevOps teams running repeatable scans across repositories

Customer Outcome

Instead of relying on diagrams alone, customers can review concrete evidence showing where sensitive fields appear, which files are likely front-end references only, which files are likely API or business-logic owners, which service files correlate to likely data-access owners, and where a Thales JDBC driver may reduce code changes.

The result is faster scoping, clearer project planning, better DBA preparation, better developer alignment, and more practical discussions about CRDP REST versus JDBC-based protection.

The scanner is built around source-code identifiers and aliases, which keeps the approach portable and suitable for future Java-based implementations as well.
